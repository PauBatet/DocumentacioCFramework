Docker Presets

CFramework provides pre-configured container architectures to streamline deployment. These presets include high-performance Caddy front-ends for TLS termination and automatic project recompilation.

1. SQLite Architecture

Ideal for development or applications with moderate persistence needs. The database is a file stored in a Docker volume shared with the application container.

CFramework sqlite_docker
  • Minimal resource footprint.
  • Persistent storage via Docker volumes.
  • Caddy handles all incoming HTTP/HTTPS traffic.

2. PostgreSQL Integrated

A self-contained environment where the PostgreSQL engine runs as a sidecar service.

CFramework postgres_docker

The application container includes a health-check dependency, ensuring the C server only boots once the Postgres engine is ready to accept connections. 

# The startup sequence automatically handles:
1. Project Recompilation
2. Database Migration Execution
3. Server Launch

3. PostgreSQL Standalone

Designed for production environments where the database is hosted on a managed service (like AWS RDS) or a dedicated external server.

CFramework postgres_standalone_docker

This preset removes the local database container. Connection parameters are passed strictly through environment variables, facilitating secure secrets management.

Caddy: The Front-end

All presets utilize Caddy. This allows the C application to focus purely on business logic while Caddy manages:

  • Automatic TLS (SSL) certificates.
  • HTTP/2 and HTTP/3 support.
  • Static file serving for the static/ directory.
Pro Tip: Use the integrated presets for CI/CD pipelines to ensure the environment exactly matches your production standalone configuration.
