Database Models

Models in CFramework are declared in C and automatically generate the necessary SQL schema, C structures, and CRUD functions.

1. Declaring Models

Define your schema using Field and ForeignKey arrays. Use the model() function to register them. 

void register_group_model(void) {
    Field user_fields[] = {
        {"name", TYPE_STRING},
        {"age", TYPE_INT},
        {"email", TYPE_STRING},
        {"group_id", TYPE_INT}
    };

    ForeignKey user_fk[] = {
        {"group_id", "Group", "id"}
    };

    Field pk_field = { "DNI", TYPE_STRING };

    // Syntax: model(name, primary_key, fields, field_count, fks, fk_count)
    model("User", &pk_field, user_fields, 4, user_fk, 1);
    
    // Passing NULL for primary_key automatically creates an "id" SERIAL/INTEGER PRIMARY KEY
    model("Group", NULL, group_fields, 2, NULL, 0);
}

TYPE_INTINTEGER
TYPE_FLOATREAL / DOUBLE PRECISION
TYPE_STRINGTEXT / VARCHAR
TYPE_BOOLBOOLEAN

2. Generated CRUD API

For every model (e.g., User), the framework generates a dedicated header and source file providing the following functions:

Create & Update

  • bool User_create(db, &obj): Inserts a new record.
  • bool User_update(db, &obj): Updates record based on Primary Key.
  • bool User_create_many(db, &list): Batch insert within a transaction.

Read & Query

  • bool User_read(db, PK, &obj): Fetches a single record.
  • bool User_read_all(db, &list): Fetches all records into a list.
  • bool User_query_unsafe(db, "where_clause", &list): Execute raw SQL filters.

Delete

  • bool User_delete(db, PK): Deletes a specific record.

3. Concurrency & Locking

When using PostgreSQL as the backend, CFramework generates locking-aware read functions to handle row-level locks during transactions. 

// Available in Postgres mode
User_read_for_update(db, "12345678X", &user);
User_read_all_for_update(db, &user_list);

These functions append FOR UPDATE to the SQL query, preventing other transactions from modifying the rows until your current transaction finishes.

4. Essential Cleanup

Because CFramework models utilize dynamic allocation for string fields, you must use the generated free functions to prevent memory leaks. 

// Single Object
User_free(&user);

// Lists
UserList_free(&list);
Note: User_read automatically calls User_free on the target object before populating it, ensuring that reusing a struct variable doesn't leak previous data.
