Core Concepts

CFramework is built on the philosophy of Compiled Efficiency. Unlike interpreted frameworks, every route, model, and view is part of a single, highly-optimized binary.

1. Database Backend Selection

CFramework is designed to be backend-agnostic. You can switch between a lightweight, file-based SQLite database or a robust PostgreSQL instance simply by modifying config.c. 

// config.c
const char *DB_BACKEND = DB_SQLITE;
// or
const char *DB_BACKEND = DB_POSTGRES;

This selection changes how the internal generator produces your Model SQL and how the binary interacts with the data layer, without changing your business logic code.

2. Request Lifecycle

When a request hits your application, it follows a strict path:

1. Front-end Proxy (Caddy)

Caddy receives the raw HTTP/HTTPS connection, handles TLS, and forwards the request to the C binary.

2. Router Dispatch

The internal router parses the URL, extracts parameters (like DNI), and executes the matching View function.

3. View Execution

Your C code interacts with the Database using the generated Models API and prepares data for the template.

4. Template Injection

The template engine performs a single-pass string substitution and sends the final HTML back through the socket.

3. The Role of Code Generation

CFramework uses a Static Schema approach. When you register a model, the framework generates:

  • .h files with type-safe C structs.
  • .c files with SQL statements optimized for your backend (SQLite/Postgres).
  • Automated Migration scripts to sync your DB with your C structures.
This eliminates the need for a runtime Reflection API or heavy ORM overhead, resulting in near-instantaneous database operations.

4. Memory Ownership

Managing memory is the only "manual" part of CFramework. To keep the framework fast and leak-free:

  • Database Reads: The _read functions use strdup() for strings. You must call _free().
  • Template Params: The framework manages the memory of tokens injected into templates if they are produced via process_html, but the View must free its own local buffers.
